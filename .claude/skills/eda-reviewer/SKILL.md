# EDA Reviewer

Audit an EDA report generated by the **dataset-analyzer** skill and return detailed, actionable feedback.
This skill runs in **completeness & quality mode**: it trusts the executor's computed numbers and instead
checks whether the report is complete, well-presented, internally consistent, and useful. It does **not**
reload the source data or recompute statistics.

## Inputs

- The report Markdown: `outputs/EDA/<dataset>_eda_report.md` (ask or infer the path if not given).
- The figures directory it references: `outputs/EDA/figures/`.
- Optionally the executive deck: `outputs/EDA/<dataset>_executive_summary.pptx`.
- Optionally `memory_eda.md` at the project root (this workstream's cross-run memory, separate from
  `memory_segmentation.md`) — check whether a claim in the report contradicts a prior run's recorded finding.

Read the report in full and list the files in `outputs/EDA/figures/` before judging.

## The rubric (embedded — this skill is self-contained)

The report should cover all 20 items across 5 phases. Items 18-20 are **conditionally applicable** and may
correctly be marked "Not applicable" when the data has no datetime/free-text columns.

- **Phase 1 — Business & Data Understanding:** (1) Project overview & objective, (2) Shape/structure +
  grain, (3) Data dictionary.
- **Phase 2 — Data Quality:** (4) Duplicates, (5) Missing values + fill rate + sentinel handling,
  (6) Type mismatches / mixed types, (7) Inconsistent categorical labels, (8) Range/domain violations,
  (9) Redundant / near-duplicate features.
- **Phase 3 — Univariate & Distribution:** (10) Numeric stats incl. extended percentiles, (11) Categorical
  stats + cardinality, (12) Distribution shape (skew/kurtosis) + charts, (13) Outlier detection (IQR/Z).
- **Phase 4 — Relationships & Target:** (14) Correlation + heatmap, (15) Target deep-dive / class balance,
  (16) Bivariate/multivariate charts, (17) Feature interactions.
- **Phase 5 — Specialized (gated):** (18) Time-based trends, (19) Time-series deep dive, (20) Text EDA.

## What to check (four dimensions)

1. **Completeness.** Is every applicable item present as its own section? Are gated items (18-20) either
   done or *explicitly* marked "Not applicable" with a reason — not silently dropped? A missing section is a
   gap; a silently dropped gated item is also a gap.

2. **Presentation.** Does each item have the expected artifact (a table, or a chart where the rubric calls
   for one)? Do the embedded image links actually resolve to files that exist in `outputs/EDA/figures/`?
   Flag any `![...](figures/x.png)` whose target file is absent. Are charts captioned?

3. **Interpretation quality (internal consistency).** Do the narrative claims agree with the report's own
   tables? Cross-check, for example:
   - a "constant column" claim ↔ a unique-count of 1 in the stats table;
   - a "perfectly balanced" / "imbalanced" target claim ↔ the reported class counts and imbalance ratio;
   - a "near-duplicate features" claim ↔ the correlation value cited (should exceed ~0.95);
   - skew/kurtosis wording ("strongly skewed", "symmetric") ↔ the numeric skew value shown;
   - the executive-summary health flags ↔ the detailed sections they summarize.
   Report any claim that contradicts the numbers in the same document. (Do not recompute from raw data.)

4. **Actionability.** Is there a plain-language summary at the top and a concrete, prioritized "next steps"
   list? Would a reader know what to fix first? Flag vague or missing guidance.

## Output

Write `outputs/EDA/<dataset>_eda_review.md` with:

1. **Overall completeness summary** — one short paragraph: how many of the applicable items are Present,
   how many Partial/Missing, and the single most important gap (if any). **No numeric grade** — a
   plain-language verdict instead (e.g. "Comprehensive and internally consistent; ready to act on").

2. **Per-item verdict table** — one row per checklist item:

   | # | Item | Verdict | Notes |
   |---|------|---------|-------|

   Verdict is one of **Present / Partial / Missing / N-A**. Notes quote or reference the specific section and
   say precisely what is missing or wrong. For N-A, confirm the report justified it.

3. **Consistency findings** — bullet list of any narrative-vs-table contradictions found (or "none found").

4. **Broken references** — list any figure links that don't resolve (or "all figures resolve").

5. **Prioritized fixes** — an ordered list the executor (or user) should act on, most important first.
   If the report is already solid, say so plainly and keep this short.

## Handoff & boundaries

- The reviewer **does not edit** the report or figures — it only produces the review file. The executor (or
  user) acts on the feedback; a full cycle is *executor → reviewer → optional executor fix pass*.
- Stay in completeness & quality mode: no data reload, no recomputation. If a number looks implausible, note
  it as "worth verifying" rather than recomputing it yourself.
- Be specific and fair: praise what's done well, and make every criticism actionable.
